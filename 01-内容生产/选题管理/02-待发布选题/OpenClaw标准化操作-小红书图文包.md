# OpenClaw标准化操作 - 小红书图文包

## 图文转化判断

- 结论：`可转化，建议做`
- 时效性：`半衰期型偏常青`
- 主模版：`方法教程型`
- 辅模版：`身份赋能型`
- 适合原因：
  - 这是典型的“高频痛点 -> 一个具体解法”的图文题
  - 用户会因为“以后要照着做”而收藏
  - 图文比视频更适合展示 `SOP 输入 -> 生成结果 -> 执行效果`
- 当前问题：
  - 原视频稿过度依赖口播情绪
  - “1个小众技巧” 这个钩子太弱
  - 中间没有清晰的截图链路，图文会显得像在空讲

## 这篇图文不要怎么做

- 不要继续用：`1个小众技巧实现 OpenClaw 标准化操作`
- 不要从“我用了两个月 OpenClaw”开始讲
- 不要大段解释模型强弱
- 不要把重点放在 `Opus 4.6 / GPT 5.4` 谁更强

这篇真正的卖点不是模型，而是：

`当 OpenClaw 经常中断、发散、不按流程做事时，先让更成熟的 agent 给它写 SOP。`

## 推荐定位

- 目标用户：已经在用 OpenClaw，且遇到过中断、发散、gateway 报错、任务做不完的人
- 内容任务：不是安利 OpenClaw，而是解决“它不稳定、不好带”的问题
- 用户带走感：看完能立刻照着做一个标准化 SOP

## 图片结构建议

- 总页数：`9-10 页`
- 纯文字页：`3-4 页`
- 操作截图页：`4-5 页`
- 结构图 / 对比页：`1-2 页`
- 推荐比例：`文字 : 图片 ≈ 45 : 55`

这篇图文的图不能少。

如果只有文字，没有截图，读者会觉得你在讲理念，不会觉得这是一个可复用的方法。

## 需要准备的图片

### 必须有

1. `OpenClaw 报错 / 中断 / 跑偏` 的真实界面截图
2. 你把任务描述交给 `Codex / Claude Code` 的输入截图
3. `Codex / Claude Code` 生成 SOP 的结果截图
4. 把 SOP 交还给 OpenClaw 后的执行截图
5. `前后对比图`

### 最好有

6. 一张 `SOP 结构示意图`
7. 一张 `适合用 / 不适合用` 的判断卡

## 截图处理要求

- 每张截图只保留一个重点，不要整屏都塞上去
- 关键位置要加框：
  - 报错位置
  - 任务要求
  - SOP 目录结构
  - 执行结果
- 截图页上方一定要加一句大字结论，不能裸放图
- 如果截图文字太小，就只截局部，不要追求完整

## 页卡结构

### 第 1 页：点击钩子

大字：

`OpenClaw 老是中断？`

`我现在都先让别的 agent 给它写 SOP`

小字：

很多人以为是模型不够强，但实际问题经常是：
- 任务描述不完整
- 执行步骤不稳定
- 约束条件没被写清楚

### 第 2 页：问题定义

大字：

`龙虾不是不会做事`

`而是你没先把流程标准化`

正文：

如果你也遇到过这些情况：
- 聊着聊着突然中断
- 做到一半开始发散
- 明明装了很多 skill 还是接不住任务
- 同一个任务每次做法都不一样

那这篇讲的不是“怎么让它更聪明”，而是“怎么让它更稳定”。

配图：
- 1 张真实报错或跑偏截图

### 第 3 页：核心方法亮相

大字：

`先别直接教 OpenClaw`

`先让更成熟的 agent 帮你写 SOP`

正文：

我现在的做法是：

先把任务目标、过程要求、交付标准、限制条件交给 `Codex / Claude Code`，
让它先产出一份更清晰的操作说明书，
再把这份 SOP 交给 OpenClaw 去执行。

这一步的作用不是替代 OpenClaw，
而是先把“模糊任务”变成“标准任务”。

### 第 4 页：类比翻译

大字：

`把 Codex 当项目经理`

`把 OpenClaw 当执行者`

正文：

如果直接让 OpenClaw 边理解边执行，
它很容易在长任务里跑偏。

但如果先让另一个更擅长规划和拆解的 agent 做前置梳理，
OpenClaw 拿到的就不是一团需求，
而是一份按步骤写好的任务说明书。

配图：
- 1 张简化结构图
- 结构：`你的需求 -> Codex / Claude Code -> SOP -> OpenClaw`

### 第 5 页：SOP 输入要写什么

大字：

`你至少要写清这 4 件事`

正文：

让另一个 agent 帮你写 SOP 时，不要只丢一句“帮我做”。

至少写清：
1. 目标：最终要交付什么
2. 过程：你希望它按什么步骤做
3. 约束：不能做什么，要避免什么
4. 输出：结果要用什么格式呈现

配图：
- 1 张你真实输入 `Codex / Claude Code` 的截图

### 第 6 页：SOP 长什么样

大字：

`一份可用 SOP，至少要有这几段`

正文：

- 任务背景
- 执行步骤
- 需要调用的工具 / skill
- 错误时怎么处理
- 结束条件是什么

如果没有“错误处理”和“结束条件”，
长任务就很容易中断或者无限发散。

配图：
- 1 张 `Codex / Claude Code` 输出 SOP 的截图

### 第 7 页：交回 OpenClaw 之后会发生什么

大字：

`它不会一下变聪明`

`但会明显更稳`

正文：

这套方法解决的不是智商问题，
是执行稳定性问题。

你会明显看到：
- 它更容易按顺序做
- 对中间状态的理解更一致
- 任务中断和跑偏会减少

配图：
- 1 张执行中的截图

### 第 8 页：前后差异

大字：

`标准化前后，差别主要在这里`

正文：

标准化前：
- 你每次都得重新解释
- 任务做到一半容易断
- 技能装了很多，但用得不稳定

标准化后：
- 需求交接更清楚
- 执行路径更固定
- 同类任务更容易复用

配图：
- 1 张前后对比卡

### 第 9 页：适合谁用

大字：

`这招更适合这 3 类人`

正文：

1. 已经在用 OpenClaw，但经常被中断搞崩的人
2. 经常做长任务、复杂任务、连续任务的人
3. 想把一类任务慢慢调成固定工作流的人

不太适合：
- 还没开始用 OpenClaw 的小白
- 只做一次性简单任务的人

## 正文

如果你的 OpenClaw 经常中断、跑偏、长任务做不完，可以先别急着换模型。

这个方法解决的不是“它够不够聪明”，而是“它能不能更稳定地把一类任务做完”。
