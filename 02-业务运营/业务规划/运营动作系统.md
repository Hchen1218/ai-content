# 运营动作系统

这是 `运营OS.md` 的运营子系统说明文件。

目标不是管理团队任务，而是管理你的账号增长动作，让系统持续回答 3 个问题：
- 这周到底在做什么运营动作
- 什么数据算做成了
- 哪些动作应该继续放大，哪些应该停

## 核心文件

| 文件 | 用途 |
|------|------|
| `📋 进行中的运营动作.md` | 所有尚未完成的运营动作卡片 |
| `✅ 已完成的运营动作.md` | 所有已归档的运营动作，也是更新策略和能力画像的唯一事实源 |
| `账号运营策略.md` | 当前平台角色、阈值、必做项、停做项 |
| `账号能力画像.md` | 从结果反推账号擅长什么、不擅长什么 |
| `周期复盘/` | 每周复盘、问题拆解、下周动作建议 |

## 运营动作卡格式

**进行中：**
```markdown
> [!task] 运营动作
> **ID：** OPS-YYYY-Www-01
> **Platform：** 抖音 / 小红书 / 双平台
> **Goal：** 拉新 / 涨粉 / 收藏 / 搜索 / 人设 / 验证
> **When：** YYYY-MM-DD ~ YYYY-MM-DD
> **What：** 动作描述
> **Content Link：** 对应内容文件路径，允许为空
> **How to measure：** 明确阈值
> **Priority：** P1 / P2 / P3
> **Status：** 待执行 / 执行中 / 待复盘
```

**已完成：**
```markdown
> [!done] 已完成运营动作
> **ID：** OPS-YYYY-Www-01
> **Platform：** 抖音 / 小红书 / 双平台
> **Goal：** 拉新 / 涨粉 / 收藏 / 搜索 / 人设 / 验证
> **When（原计划）：** YYYY-MM-DD ~ YYYY-MM-DD
> **When（实际完成）：** YYYY-MM-DD
> **What：** 动作描述
> **How to measure：** 原阈值
> **Actual Result：** 实际结果
> **Decision：** 放大 / 继续 / 停止 / 改平台 / 改形式
> **复用结论：** 一句话总结
```

## 固定枚举

### Goal 只允许以下 6 类
- `拉新`
- `涨粉`
- `收藏`
- `搜索`
- `人设`
- `验证`

### Decision 只允许以下 5 类
- `放大`
- `继续`
- `停止`
- `改平台`
- `改形式`

## 工作流

```text
创建运营动作
  ↓
📋 进行中的运营动作.md
  ↓ 数据导入后自动生成结果草稿
生成周运营复盘
  ↓ 人工确认
完成运营动作
  ↓
✅ 已完成的运营动作.md
  ↓ 定期运行
更新账号策略 / 更新账号能力
```

## 运行原则

### 1. 动作先于内容
不要先发内容再想这条是干什么的。先写动作，再决定内容怎么做。

### 2. 一个动作只回答一个核心问题
例如：
- 这个 API 题材是否适合抖音拉新
- 这个教程是否更适合小红书图文搜索流量
- 这个标题公式是否能提高点击率

### 3. 没结果不沉淀
只有进入 `✅ 已完成的运营动作.md` 的内容，才有资格反推策略和能力。

### 4. 系统只做半自动
系统负责：
- 导入数据后生成结果草稿
- 周复盘输出
- 策略更新建议

系统不负责：
- 自动发帖
- 自动评论运营
- 自动把动作从进行中改成已完成

## 当前默认平台分工
- 抖音：拉新 + 放大
- 小红书：搜索承接 + 收藏沉淀

## 关联入口
- 项目总入口：`运营OS.md`
- 平台策略：`账号运营策略.md`
- 能力画像：`账号能力画像.md`
- 周复盘：`周期复盘/`
